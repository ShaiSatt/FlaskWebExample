✏️ פקודת UPDATE — תחביר

פקודת UPDATE משנה ערכים בשורות קיימות בטבלה. לא יוצרת שורות חדשות — רק מעדכנת:

  SQL — מבנה UPDATE
UPDATE users
SET    password = ?,
       email    = ?,
       phone    = ?
WHERE  username = ?
UPDATE users
שם הטבלה שרוצים לעדכן.
SET password = ?, email = ?, phone = ?
אילו עמודות לעדכן ולמה. עמודות שלא מוזכרות — נשארות ללא שינוי.
WHERE username = ?
קריטי! מגדיר אילו שורות לעדכן. בלי WHERE — כל השורות בטבלה יתעדכנו!

⚠️ WHERE — כלל ברזל

❌ בלי WHERE — אסון!
UPDATE users
SET password = 'abc'

-- כל המשתמשים מקבלים
-- את אותה סיסמה!
-- אין דרך לבטל!
✅ עם WHERE — בטוח
UPDATE users
SET password = 'abc'
WHERE username = 'alice'

-- רק alice מתעדכנת
-- שאר המשתמשים
-- לא נפגעים
🚨
כלל ברזל: UPDATE ו-DELETE תמיד עם WHERE. לפני הרצה — שאלו: "על אילו שורות זה ישפיע?"

🔄 route ה-/update — GET ו-POST

דף העדכון פועל בשני שלבים — GET לטעינת הפרטים הנוכחיים, POST לשמירת השינויים:

  main.py — route /update
@app.route('/update', methods=['GET', 'POST'])
def update():
    if current_user == 'Guest':
        return redirect(url_for('login'))

    if request.method == 'GET':
        conn = sqlite3.connect('users.db')
        c = conn.cursor()
        c.execute("SELECT * FROM users WHERE username = ?", (current_user,))
        user_details = c.fetchone()
        conn.close()
        return render_template('update.html', user_details=user_details)

    if request.method == 'POST':
        new_password = request.form['new_password']
        new_email    = request.form['new_email']
        new_phone    = request.form['new_phone']
        conn = sqlite3.connect('users.db')
        c = conn.cursor()
        c.execute("""UPDATE users
                      SET password=?, email=?, phone=?
                      WHERE username=?""",
                 (new_password, new_email, new_phone, current_user))
        conn.commit()
        conn.close()
        flash("הפרטים עודכנו בהצלחה!", "success")
        return redirect(url_for('update'))

🔍 GET — טעינת הפרטים הנוכחיים

SELECT * WHERE username = current_user
שולפים את הפרטים הנוכחיים של המשתמש המחובר ושולחים לתבנית בשם user_details.
render_template('update.html', user_details=...)
ה-tuple מגיע לתבנית. שם ניגש אליו עם user_details[0], user_details[1] וכו׳.

🔍 POST — שמירת השינויים

UPDATE ... SET ... WHERE username = current_user
מעדכנים רק את השורה של המשתמש המחובר. current_user כ-WHERE מבטיח שמשתמש לא יכול לערוך אחרים.
redirect(url_for('update'))
אחרי שמירה — מפנים חזרה לאותו דף (GET) כדי לראות הפרטים המעודכנים. הודעת Flash תוצג.

🎨 update.html — הקובץ המלא

הטופס מציג את הפרטים הנוכחיים בשדות כך שהמשתמש יכול לשנות רק מה שרוצה:

  templates/update.html
{% extends "base.html" %}

{% block title %} update {% endblock %}

{% block content %}
<h2>Update User Details</h2>

<!-- טופס א׳: עדכון -->
<form action="/update" method="post">
  <label> <b> Username: {{ user_details[0] }} </b> </label><br>

  <label for="new_password">New Password: </label>
  <input type="text" name="new_password" value= {{ user_details[1] }} ><br>
  <label for="new_email">New Email: </label>
  <input type="email" name="new_email" value= {{ user_details[2] }} ><br>
  <label for="new_phone">New Phone: </label>
  <input type="text" name="new_phone" value= {{ user_details[3] }}><br><br>

  <input type="submit" value="Update Details">
</form>

<!-- טופס ב׳: מחיקה — action שונה! -->
<form action="/delete" onclick="return performAction()" method="post">
    <button type="submit">Delete Me!!</button>
</form>

<script>
    function performAction() {
        return window.confirm("בטוח שרוצה למחוק את המשתמש שלך לצמיתות?");
    }
</script>
{% endblock %}

🔍 ניתוח התבנית

{{ user_details[0] }}
username מוצג ב-<label> בלבד — לא ב-<input>. הסיבה: username הוא PRIMARY KEY, לא ניתן לשינוי.
value= {{ user_details[1] }}
ממלא את השדה עם הערך הנוכחי מה-DB. המשתמש רואה מה יש וכותב מעליו אם רוצה לשנות.
action="/delete"
הטופס השני שולח POST ל-/delete — שונה מהטופס הראשון שהולך ל-/update. שני טפסים נפרדים לחלוטין!
onclick="return performAction()"
לפני שליחה — מריץ JavaScript. אם מחזיר false (לחצו "ביטול") — הטופס לא נשלח.

🗑️ פקודת DELETE ו-route המחיקה

DELETE מוחקת שורות מהטבלה. בפרויקט — משתמש יכול למחוק את עצמו בלבד:

  SQL — מבנה DELETE
DELETE FROM users
WHERE username = ?

-- בלי WHERE: DELETE FROM users ← מוחק את כל הטבלה!
  main.py — route /delete
@app.route('/delete', methods=['POST'])
def delete():
    global current_user
    if current_user == 'Guest':
        return redirect(url_for('login'))

    conn = sqlite3.connect('users.db')
    c = conn.cursor()
    c.execute("DELETE FROM users WHERE username = ?", (current_user,))
    conn.commit()
    conn.close()

    current_user = 'Guest'     ← חובה לנתק אחרי מחיקה!
    flash("המשתמש נמחק", "success")
    return redirect(url_for('home'))

🔍 ניתוח route המחיקה

DELETE FROM users WHERE username = ?
מוחקים רק את השורה של המשתמש המחובר. ה-? ממולא ב-current_user.
current_user = 'Guest'
חשוב מאוד! אחרי מחיקה — חייבים לנתק. אחרת current_user יחזיק שם שלא קיים ב-DB וכל פעולה תגרום שגיאה.
methods=['POST']
מקבל POST בלבד — מונע מחיקה מקרית בגישה ישירה לכתובת /delete בדפדפן.

💬 window.confirm() — אישור מחיקה

לפני מחיקה מוצג חלון אישור JavaScript. כך זה עובד:

  הקוד ב-update.html
<form action="/delete"
      onclick="return performAction()"
      method="post">
    <button type="submit">Delete Me!!</button>
</form>

<script>
    function performAction() {
        return window.confirm("בטוח שרוצה למחוק את המשתמש שלך לצמיתות?");
    }
</script>

🔍 זרימת האישור

1
לחיצה על "Delete Me!!"
ה-onclick על הטופס מופעל לפני שליחת הבקשה לשרת
2
performAction() רץ — חלון קופץ
window.confirm() פותח חלון בדפדפן עם כפתורי "אישור" ו-"ביטול"
3א
לחיצה "אישור" ← מחיקה
confirm() מחזיר true → הטופס נשלח → POST ל-/delete
3ב
לחיצה "ביטול" ← כלום לא קורה
confirm() מחזיר false → הטופס לא נשלח → הכל נשאר

🧩 בדקו את עצמכם!

שאלה 1 — מה יקרה אם נריץ UPDATE users SET password='abc' בלי WHERE?
רק המשתמש המחובר יתעדכן
כל המשתמשים בטבלה יקבלו סיסמה 'abc' — פעולה בלתי הפיכה!
SQLite יזרוק שגיאה כי חסר WHERE
כלום — צריך commit קודם
שאלה 2 — למה route /update מגדיר methods=['GET','POST'] ואילו /delete רק methods=['POST']?
DELETE לא עובד עם GET
/update צריך GET לטעינת הטופס ו-POST לשמירה. /delete לא מציג דף — POST בלבד מונע מחיקה מקרית בגישה ישירה לכתובת
הגדרה טכנית בלבד, אין הבדל מעשי
שאלה 3 — למה username מוצג ב-<label> ולא ב-<input>?
Jinja2 לא יכול לשים ערך ב-input
username הוא PRIMARY KEY — לא ניתן לשינוי. label מדגיש שהוא קריאה בלבד
label נראה יפה יותר
שאלה 4 — מה קורה כשלוחצים "ביטול" ב-window.confirm()?
נשלח POST ל-/delete בכל מקרה
מופיעה הודעת Flash מ-Flask
confirm() מחזיר false → onclick מחזיר false → הטופס לא נשלח בכלל
שאלה 5 — מדוע כותבים current_user = 'Guest' אחרי המחיקה?
כדי לפנות זיכרון
Flask דורש זאת לפני redirect
המשתמש לא קיים יותר ב-DB. בלי איפוס — current_user יכיל שם שנמחק וכל SELECT/UPDATE עליו יגרום שגיאות

🏁 סיכום — CRUD מושלם!

  • UPDATE users SET col=? WHERE username=? — עדכון שורה קיימת
  • DELETE FROM users WHERE username=? — מחיקת שורה
  • WHERE תמיד! — בלעדיו UPDATE/DELETE פועלים על כל הטבלה
  • route /update: GET = טעינת פרטים, POST = שמירת שינויים
  • username ב-label בלבד — PRIMARY KEY לא ניתן לשינוי
  • route /delete: methods=['POST'] בלבד
  • אחרי מחיקה — current_user = 'Guest' מיד!
  • window.confirm() — הגנה ב-JS לפני פעולות הרסניות
