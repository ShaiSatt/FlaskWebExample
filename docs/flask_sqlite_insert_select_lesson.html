📦 מה זה Tuple?

SQLite מחזיר נתונים בצורה של tuple — אחד מסוגי הנתונים הבסיסיים ב-Python. חשוב להכיר אותו לפני שנמשיך:

🔑 הגדרה

Tuple הוא רצף סדור ובלתי ניתן לשינוי של ערכים. נכתב עם סוגריים עגולים (), האיברים מופרדים בפסיקים:

# tuple של משתמש מהDB:
user = ("alice", "pass123", "alice@school.ac.il", "050-111")

# גישה לפי אינדקס — בדיוק כמו list:
user[0]  → "alice"
user[1]  → "pass123"
user[2]  → "alice@school.ac.il"
user[3]  → "050-111"
Tuple ()
user = ("alice", "pass", "a@b.com")

# קריאה — מותר ✓
print(user[0])  → "alice"

# שינוי — אסור! ✗
user[0] = "bob"
→ TypeError: 'tuple' object
   does not support item
   assignment
List []
users = ["alice", "bob", "carol"]

# קריאה — מותר ✓
print(users[0])  → "alice"

# שינוי — מותר ✓
users[0] = "dave"
→ עובד בלי בעיה

🗃️ למה SQLite מחזיר tuple ולא list?

שורת DB היא קבועה — לא אמורים לשנות אותה ישירות בתוצאה. tuple מתאים כי הוא מייצג נתון קריאה-בלבד.

fetchone()
מחזיר tuple אחד, כמו: ("alice", "pass123", "alice@school.ac.il", "050-111") — או None אם לא נמצא דבר.
fetchall()
מחזיר list של tuples: [("alice", ...), ("bob", ...), ...] — רשימה שאפשר לעבור עליה בלולאה.
📦tuple vs list — השוואה
תכונהTuple ()List []
יצירה ("a", "b", "c") ["a", "b", "c"]
גישה לפי אינדקס ✓ כן ✓ כן
שינוי ערך ✗ לא ✓ כן
הוספת איברים ✗ לא ✓ כן
מה SQLite מחזיר? ✓ שורה = tuple רשימה = list של tuples
💡
בקיצור: tuple = list שלא אפשר לשנות. ב-DB שלנו — כל שורה = tuple. user[0] = username, user[1] = password, לפי סדר העמודות שהגדרנו ב-CREATE TABLE.

✍️ INSERT INTO — הוספת משתמש חדש

פונקציית insert_user() מוסיפה רשומה חדשה לטבלה. היא נקראת מתוך route ה-signup כשמשתמש נרשם:

  main.py — insert_user()
def insert_user(username, password, email, phone):
    conn = sqlite3.connect('users.db')
    c = conn.cursor()

    # שלב א׳: בדיקה — האם username כבר קיים?
    c.execute("SELECT * FROM users WHERE username = ?", (username,))
    existing = c.fetchone()
    if existing:
        conn.close()
        return False           ← משתמש כבר קיים

    # שלב ב׳: הוספה — username חדש
    c.execute("INSERT INTO users VALUES (?, ?, ?, ?)",
             (username, password, email, phone))
    conn.commit()
    conn.close()
    return True               ← הצלחה

🔍 ניתוח: שני שלבים לפני INSERT

SELECT ... WHERE username = ?
בדיקה ראשונה: לפני שמוסיפים — בודקים אם username זה כבר קיים. PRIMARY KEY היה זורק Exception; כך אנחנו שולטים בתגובה.
fetchone()
מחזיר tuple אם מצא תוצאה, או None אם לא. אם קיבלנו תוצאה — username תפוס, נחזיר False.
INSERT INTO users VALUES (?, ?, ?, ?)
ארבעה ? = ארבעה ערכים לפי סדר העמודות: username, password, email, phone. SQLite ממלא אותם מה-tuple.
commit() → return True
שומרים לדיסק ומחזירים True. ה-route יוכל להציג הודעת הצלחה ולהפנות לדף login.

🛡️ שאילתות עם פרמטרים — למה ??

שימו לב לסימן ? בשאילתות. זה אחד הנושאים החשובים ביותר בעבודה עם DB:

❌ שרשור מחרוזות — מסוכן!
# נראה הגיוני...
query = "SELECT * FROM users " \
      + "WHERE username = '" \
      + username + "'"

# אם username = "' OR '1'='1"
# השאילתה הופכת ל:
# WHERE username = ''
#    OR '1'='1'
# ← מחזיר את כל המשתמשים!
✅ Parameterized — בטוח
# ? = placeholder
c.execute(
  "SELECT * FROM users "
  "WHERE username = ?",
  (username,)   ← tuple!
)
# SQLite מטפל בערך בבטחה
# גם עם תווים מיוחדים
# ← מוגן מ-SQL Injection

⚠️ למה (username,) עם פסיק?

הפרמטר השני של execute חייב להיות tuple. Python מבחינה בין ביטויים בסוגריים ל-tuple:

(username)    ← ביטוי בסוגריים — לא tuple!
(username,)   ← tuple עם איבר אחד — נכון ✓

(username, password)  ← tuple של שניים — בסדר ✓
⚠️
SQL Injection הוא אחד מסוגי ההתקפות הנפוצים על אתרים. שאילתות עם ? מגינות עלינו ממנו אוטומטית — SQLite מטפל בתווים מיוחדים לפני הרצת השאילתה.

📝 route ה-Signup — כל הזרימה

ה-route מקבל נתונים מהטופס, קורא ל-insert_user(), ומגיב בהתאם:

  main.py — @app.route('/signup')
@app.route('/signup', methods=['GET', 'POST'])
def signup():
    if request.method == 'POST':
        username = request.form['username']
        password = request.form['password']
        email    = request.form['email']
        phone    = request.form['phone']

        success = insert_user(username, password, email, phone)

        if success:
            flash("נרשמת בהצלחה!", "success")
            return redirect(url_for('login'))
        else:
            flash("שם המשתמש כבר תפוס", "error")

    return render_template('signup.html', user=current_user)

🔍 ניתוח הזרימה

request.form['username']
שולפים את הערכים שהמשתמש הקליד בטופס. שם השדה חייב להתאים ל-name של <input> ב-HTML.
insert_user(...)
קוראים לפונקציה שכתבנו. היא מחזירה True אם הצליחה, False אם username תפוס.
flash + redirect → login
הצלחה — הודעת Flash "נרשמת בהצלחה!" ומפנים לדף login כדי שיתחבר עם הפרטים החדשים.
flash "שם המשתמש כבר תפוס"
כישלון — מציגים הודעת שגיאה. הקוד ממשיך לסוף ומחזיר את עמוד signup שוב עם ההודעה.
💡
הזרימה המלאה: משתמש שולח טופס ← POST מגיע ← שולפים ערכים ← insert_user() ← הצלחה? redirect לlogin — כישלון? signup שוב + הודעה

🔑 SELECT לצורך Login — fetchone()

ב-login לא רוצים את כל המשתמשים — רק לבדוק אם username + password תואמים:

  main.py — @app.route('/login')
@app.route('/login', methods=['GET', 'POST'])
def login():
    global current_user
    message = None
    if request.method == 'POST':
        username = request.form['username']
        password = request.form['password']
        conn = sqlite3.connect('users.db')
        c = conn.cursor()
        c.execute("SELECT * FROM users WHERE username = ?", (username,))
        user = c.fetchone()
        conn.close()
        if user:                         ← משתמש נמצא?
            if user[1] == password:      ← סיסמה נכונה?
                current_user = user[0]
                flash(" !נכנסת בהצלחה", "success")
                return redirect(url_for('home'))
            else:
                message = "סיסמה שגויה"
        else:
            message = "שם משתמש לא קיים"
    return render_template('login.html', message=message, user=current_user)

🔍 מבנה tuple מ-fetchone()

כשה-SELECT מחזיר תוצאה, fetchone() מחזיר tuple לפי סדר העמודות בטבלה:

מה fetchone() מחזיר
משתנהערך לדוגמהמה זה?
user[0]"alice"username
user[1]"pass123"password — משווים אליו
user[2]"alice@school.ac.il"email
user[3]"050-1234567"phone
if user:
tuple הוא True, None הוא False. בדיקה ראשונה: האם username בכלל קיים?
user[1] == password
בדיקה שנייה: האם הסיסמה שהוקלדה תואמת לסיסמה שמורה ב-DB. user[1] = עמודת password.
💡
שתי הודעות שגיאה שונות: "שם משתמש לא קיים" vs "סיסמה שגויה". שתי בדיקות נפרדות מאפשרות חוויית משתמש טובה יותר.

👥 SELECT * — שליפת כל המשתמשים

דף users מציג את כל המשתמשים. כאן משתמשים ב-fetchall() שמחזיר רשימה של tuples:

  main.py — @app.route('/users')
@app.route('/users')
def show_users():
    conn = sqlite3.connect('users.db')
    c = conn.cursor()
    c.execute("SELECT * FROM users")
    users = c.fetchall()
    conn.close()
    return render_template('users.html', users=users, user=current_user)
fetchone() — שורה בודדת
# מחזיר tuple אחד:
("alice","pass","a@b.com","050")
# או None אם לא נמצא
# שימוש: login
fetchall() — כל השורות
# מחזיר list של tuples:
[("alice","pass","a@b.com","050"),
 ("bob","1234","b@c.com", None),
 ...]
# שימוש: users page

מעבירים את הרשימה לתבנית. שם המשתנה (users) חייב להתאים לשם שבו נשתמש ב-Jinja2.

🎨 Jinja2 — הצגת הנתונים ב-users.html

עכשיו נסתכל על התבנית — איך מציגים את רשימת המשתמשים שקיבלנו מה-DB:

  templates/users.html
{% extends "base.html" %}

{% block content %}
<h2>User List</h2>
<table border="1">
    <tr>
        <th>Username</th><th>Password</th>
        <th>Email</th><th>Phone</th>
    </tr>
    {% for user in users %}
    <tr>
        <td>{{ user[0] }}</td>  <!-- username -->
        <td>{{ user[1] }}</td>  <!-- password -->
        <td>{{ user[2] }}</td>  <!-- email    -->
        <td>{{ user[3] }}</td>  <!-- phone    -->
    </tr>
    {% endfor %}
</table>
{% endblock %}

🔍 ניתוח התבנית

{% for user in users %}
לולאת Jinja2 על רשימת ה-tuples מ-fetchall(). בכל איטרציה user = tuple של שורה אחת.
{{ user[0] }}
גישה לאינדקס 0 ב-tuple = username. הסדר לפי CREATE TABLE: 0=username, 1=password, 2=email, 3=phone.
{% endfor %}
סגירת הלולאה. כל מה שבין for ל-endfor חוזר עבור כל שורה בטבלה.

📋 login.html — הצגת הודעת שגיאה

בדף login ה-template מציג הודעה מותנית — רק אם קיבלנו message מה-route:

  templates/login.html
{% block content %}
<h2>Login Page</h2>
{% if message %}
    <p class="error-message">{{ message }}</p>
{% endif %}
<form method="post">
    <input type="text"   name="username">
    <input type="text"   name="password">
    <input type="submit" value="Login">
</form>
{% endblock %}
{% if message %}
מציג את ה-<p> רק כשיש הודעה שגיאה. כשה-route שולח message=None בGET — הפסקה לא מופיעה.
💡
שימו לב לשמות: name="username" ב-HTML חייב להתאים ל-request.form['username'] ב-Python. ואת המשתנה message שולחים מה-route ב-render_template(..., message=message).

שאלה 1 — מדוע insert_user() מבצעת SELECT לפני ה-INSERT?
כי SELECT מהיר יותר מ-INSERT
כדי לבדוק שה-username לא קיים — PRIMARY KEY ימנע כפילות ויזרוק Exception, אבל כך אנחנו שולטים בתגובה ומחזירים False מסודר
כי INSERT לא עובד בלי SELECT לפניו
כדי לבדוק שהחיבור ל-DB פועל
שאלה 2 — מה ההבדל בין fetchone() לבין fetchall()?
fetchone() מהיר יותר, fetchall() מדויק יותר
fetchone() מחזיר tuple בודד (או None), fetchall() מחזיר רשימה של כל ה-tuples
fetchone() לשאילתות SELECT, fetchall() לשאילתות INSERT
אין הבדל — שניהם מחזירים אותו דבר

כי SQL דורש פסיק אחרי כל ערך
כי Python דורשת tuple כפרמטר, ו-(username) זה רק סוגריים רגילים — הפסיק הופך אותו ל-tuple אמיתי

שאלה 5 — ב-login מבצעים if user: ואז if user[1] == password:. למה שתי בדיקות ולא אחת?

כי צריך לבדוק שני דברים בנפרד: (1) האם username בכלל קיים, (2) אם כן — האם הסיסמה נכונה. בלי הבדיקה הראשונה user[1] על None יקרוס

🏁 סיכום הלומדה

  • INSERT INTO users VALUES (?, ?, ?, ?) — מוסיף שורה חדשה
  • לפני INSERT — בודקים כפילות עם SELECT + fetchone()
  • סימן ? + tuple — מגן מ-SQL Injection
  • (username,) עם פסיק — יוצר tuple של איבר אחד ב-Python
  • fetchone() — tuple בודד או None (שימוש: login)
  • fetchall() — רשימה של כל ה-tuples (שימוש: users page)
  • גישה: user[0]=username, user[1]=password, user[2]=email, user[3]=phone
  • ב-Jinja2: {% for user in users %} על תוצאת fetchall()
  • ב-login: שתי בדיקות נפרדות — קיום username ואז נכונות סיסמה

🚀 משימת המשך

1. שנו את users.html כך שלא יציג את עמודת הסיסמה — מידע רגיש לא צריך להיות גלוי. רמז: הסירו את השורה עם user[1]


2. הוסיפו ב-users.html הצגת מספר המשתמשים הרשומים. Jinja2 תומך ב-{{ users|length }}


3. אתגר: שנו את שאילתת ה-SELECT ב-show_users() כך שתחזיר רק username ו-email: SELECT username, email FROM users — ועדכנו את ה-template בהתאם
